Features

Everything Repollect does to turn scattered project history into connected, queryable knowledge.

Multi-Source Knowledge Graph

Repollect ingests data from multiple platforms into a single hybrid graph-vector knowledge store powered by Cognee. Every issue, PR, Discord message, and Notion page becomes a node in a richly connected graph.

  • GitHub — Issues, PRs (open + rejected), review comments, discussions, releases, repository files (README, CONTRIBUTING, CHANGELOG)
  • Discord — Server channels, threads, forum posts, messages, text attachments
  • Notion — Pages, databases, nested blocks, rich text, comments
  • Local Files — PDF, Markdown, TXT uploads

Relationships are preserved across silos: a PR that fixed a bug discussed in Discord and documented in Notion becomes a single traversable path in the graph.

Incremental Synchronization

After the initial ingestion, Repollect only pulls new or updated data since the last sync timestamp. This makes subsequent syncs lightning-fast and cost-efficient.

  • Tracks last_synced_at per project
  • Background jobs run asynchronously with status tracking
  • Manual sync button available in the UI
  • New data merges into the existing dataset

Auto-Review Scheduler

A background worker continuously polls GitHub for newly opened Issues and Pull Requests. When it detects new activity:

  • Queries the knowledge graph for historical context related to the touched code/topic
  • Generates a helpful AI review using the LLM
  • Automatically posts the review as a GitHub comment

Configurable via AUTO_REVIEW=True and REVIEW_TIME_MIN=30 environment variables.

Smart Chat Interface

Talk to your knowledge graph with plain English and specialized slash commands:

  • /issue — Create a new GitHub issue directly from chat
  • /Notion — Create a new Notion page
  • iss{N} — Fetch and discuss a specific issue (e.g., iss54 why was this closed?)
  • pr{N} — Fetch and discuss a Pull Request
  • diff{N} — AI-powered explanation of a PR's code changes
  • /contributors — List all contributors with commit counts
  • @username — Get a summary of a specific user's contributions and activity

Two-Phase Ingestion

Repollect separates fetching from processing to minimize LLM costs:

  • Phase 1 — Stage: Each source fetches data and converts it to DataItem objects. Items are deduplicated by SHA-256 content hash and buffered via cognee.add(). No LLM or embedding work happens here.
  • Phase 2 — Cognify: A single cognee.cognify() call processes everything in the buffer — running embeddings, building the knowledge graph, and extracting entities and relations.

This means fetching is fast and LLM costs are incurred exactly once per batch.

Content Deduplication

Every piece of content is hashed using SHA-256 before storage. If the same content appears in multiple sources (e.g., a PR discussion that's also referenced in Notion), it's stored once with cross-source references.

Project Management

Create, delete, and list projects. Each project maps to one GitHub repository (owner/repo) and stores per-platform API tokens:

  • GitHub PAT (mandatory)
  • Notion Integration Token (optional)
  • Discord Bot Token (optional)

REST API

Full REST API under /api/v1 for programmatic access. Endpoints for project CRUD, ingestion, sync, chat, file upload, and health checks. Auto-generated docs at /docs when the server is running.
