Architecture

A three-layer GraphRAG architecture built on Cognee's hybrid graph-vector knowledge store.

Problem

When you join an open source project or start at a new company, the hardest knowledge to find is the context that was never written down properly: architectural decisions, rejected approaches, and historical discussions. This context is buried across thousands of scattered GitHub issues, PRs, Discord threads, changelogs, and meeting notes.

Solution

Repollect ingests a project's entire public history into a hybrid graph-vector knowledge store (powered by Cognee), then makes that knowledge queryable via a REST API and Web UI. New contributors ask questions in plain English and get answers grounded in actual project history — not hallucinations.

1. Ingestion Layer

Each source (GitHub, Discord, Notion, Files) has a dedicated ingestor class that fetches data via HTTP APIs and converts it into DataItem objects. All sources are fetched concurrently using asyncio.gather().

Items are deduplicated by SHA-256 content hash and staged into Cognee's buffer via cognee.add(). No LLM or embedding work happens during this phase — it's purely data collection and staging.

2. Cognee Core Layer

A single cognee.cognify() call processes all buffered data:

  • Generates embeddings for semantic search
  • Extracts entities (files, PRs, people, decisions, bugs)
  • Maps relationships between entities
  • Builds a hybrid graph-vector index for querying

3. Query Layer

Users interact with the knowledge graph through multiple interfaces:

  • REST API — FastAPI endpoints for programmatic access
  • Web UI — Vite + React chat interface with slash commands

Queries use cognee.search() for semantic similarity combined with graph traversal to find connected entities.

Cognee API Usage

Repollect wraps Cognee's core APIs in utility modules at app/utils/remember.py:

Cognee API Where Used Description
add() Ingestion pipeline Stage content into buffer (no LLM cost)
cognify() Ingestion pipeline Process buffer into embeddings + graph
search() Chat & API queries Semantic search + graph traversal
forget() Project deletion Prune datasets from the graph

Data Flow

Here's what happens when a user triggers ingestion for a project:

  1. Fetch: All sources are queried concurrently using asyncio.gather(). Each ingestor produces a list of DataItem objects.
  2. Deduplicate: Items are hashed (SHA-256). New items pass through; duplicates are counted and skipped.
  3. Stage: Unique items are buffered via cognee.add().
  4. Cognify: A single cognee.cognify() call processes the entire buffer.
  5. Query: Users can now search the graph via the API or chat interface.
Two-Phase Design: Separating fetch from processing means LLM costs are incurred exactly once per batch, and individual source failures don't block others.
